01

The Big Picture

Polaris has an AI chat system where users converse with an LLM that can call tools to read and modify their workspace. This PR extends that system to reach outside the workspace—into GitHub—so the AI can list repos, search issues, create issues, and assign work to Copilot, all from within a chat conversation.

The architecture has four distinct layers, each with a single responsibility:

Architecture Layers
UI Layer
IntegrationsSettingsContent SettingsDialog (Integrations tab) WorkOS Pipes Widget
API Layer
GET /integrations/connections POST /chat/trigger
Service Layer
ServerIntegrationService IIntegrationService githubApi.ts (githubFetch)
AI Layer
6 GitHub tools 2 GitHub skills ToolRegistry SkillRegistry StreamingHandler

The key insight: the AI doesn't know about OAuth, tokens, or WorkOS. It just calls tools like searchGithubIssues and gets back structured results. All the authentication complexity is hidden behind a single abstraction: IIntegrationService.

Key Insight This is the first external integration in Polaris's AI system. Every prior tool (searchNodes, createNodes, etc.) operated on local workspace data. The GitHub integration establishes the pattern for how all future external services will connect.
02

OAuth & Token Management

The hardest part of any third-party integration is authentication. Users need to grant Polaris access to their GitHub account, and the system needs to securely manage access tokens that expire and need refreshing. Polaris delegates all of this to WorkOS Pipes.

What is WorkOS Pipes?

WorkOS Pipes is an OAuth-as-a-service layer. Instead of implementing GitHub's OAuth flow yourself (redirect URIs, PKCE, token storage, refresh logic, scope management), you tell WorkOS "I need GitHub access for this user" and it handles everything. You get back an access token on demand.

The IIntegrationService Interface

Rather than coupling the AI system directly to WorkOS, the PR introduces a clean abstraction: 

integrations/types.ts
export interface IIntegrationService {
  /** Check if a provider is connected for the current user */
  isConnected(provider: string): Promise<boolean>;

  /** Get an access token for a connected provider */
  getAccessToken(provider: string): Promise<
    { token: string } | { error: string }
  >;
}

Two methods. That's the entire contract. The provider parameter is a string like "github" or "slack"—this makes the interface generic enough to support any future OAuth provider.

The Server Implementation

On the server, ServerIntegrationService implements this interface using the WorkOS Node SDK: 

ServerIntegrationService.ts
export class ServerIntegrationService implements IIntegrationService {
  constructor(
    private workos: WorkOS,
    private userId: string,
    private organizationId?: string,
  ) {}

  async getAccessToken(provider: string) {
    const result = await this.workos.pipes.getAccessToken({
      provider,
      userId: this.userId,
      organizationId: this.organizationId ?? null,
    });
    if (!result.active) {
      return { error: result.error ?? "not_connected" };
    }
    return { token: result.accessToken.accessToken };
  }
}
Pattern The service is scoped to a specific user at construction time. Each HTTP request creates a new instance with that request's authenticated user. This means the token you get is always for the right user, and there's no risk of leaking tokens between requests.

Tokens are never stored by Polaris. Every tool invocation fetches a fresh token from WorkOS, which handles refresh logic internally. If the user has revoked access or the token is expired beyond recovery, the tool gets back a clear error that it can surface to the user.

03

The GitHub API Layer

Every GitHub tool ultimately calls the same function: githubFetch. This is the narrow waist of the entire integration—a single authenticated HTTP client for the GitHub REST API. 

githubApi.ts
export async function githubFetch<T>(
  getToken: GetGithubToken | undefined,
  path: string,
  options: { method?: string; body?: unknown } = {},
): Promise<{ data: T } | { error: ToolResult }>

Three-Stage Error Handling

The function handles three distinct failure modes, each with a user-friendly message:

Stage Condition Message to AI
No Service getToken is undefined — server not configured "GitHub integration is not available. The integration service is not configured on this environment."
Not Connected User hasn't linked their GitHub account "Please connect your GitHub account in Settings > Integrations"
API Error GitHub returns a non-200 status Parses GitHub's error JSON and surfaces the message field
Design Detail The return type is a discriminated union: { data: T } | { error: ToolResult }. Every caller does if ("error" in result) return result.error. This makes error handling consistent and impossible to forget—TypeScript won't let you access .data without checking.

Security: Input Validation

The repoSchema prevents path injection in GitHub API URLs: 

export const repoSchema = z.string().regex(
  /^[a-zA-Z0-9._-]+\/[a-zA-Z0-9._-]+$/,
  "Must be in owner/repo format"
);

Without this, an LLM could potentially construct a repo value like ../../admin/secrets that would change the meaning of the API URL. The regex ensures only valid owner/repo strings pass through.

Token Scoping: getGithubToken()

Rather than passing the full IIntegrationService to every GitHub function, the PR introduces a scoping function: 

export function getGithubToken(context: ToolContext): GetGithubToken | undefined {
  if (!context.integrationService) return undefined;
  return () => context.integrationService.getAccessToken("github");
}
Pattern: Scoped Token Getters This creates a closure that can only request GitHub tokens. The tool code never sees the raw integration service, so it can't accidentally request tokens for other providers. A Slack integration would have its own getSlackToken() with the same pattern.
04

AI Tools: The Six Operations

The integration exposes six tools to the LLM. Each follows the exact same PolarisTool interface that all existing Polaris tools use—the LLM doesn't know or care that these tools call an external API.

Tool GitHub API Purpose
listGithubRepos GET /user/repos or /search/repositories Show the user's repos, optionally filtered by query
listGithubLabels GET /repos/{owner}/{repo}/labels List available labels for intelligent issue tagging
searchGithubIssues GET /search/issues Search issues with query, repo, state, label filters
readGithubIssue GET /repos/{o}/{r}/issues/{n} + /comments Read a specific issue with all its comments
createGithubIssue POST /repos/{owner}/{repo}/issues Create a new issue with title, body, labels, assignees
updateGithubIssue PATCH /repos/{o}/{r}/issues/{n} Update issue fields (title, body, state, labels, assignees)

Anatomy of a Tool

Every tool has four components. Here's readGithubIssue as a representative example:

readGithubIssue.ts
// 1. Input schema — validated by Zod before execution
const inputSchema = z.object({
  repo: repoSchema.describe("Repository in owner/repo format"),
  issueNumber: z.number().describe("Issue number to read"),
});

// 2. Executor — the actual logic
async function executor(input, context): Promise<ToolResult> {
  const token = getGithubToken(context);

  // Parallel fetch: issue details + comments
  const [issueResult, commentsResult] = await Promise.all([
    githubFetch(token, `/repos/${repo}/issues/${issueNumber}`),
    githubFetch(token, `/repos/${repo}/issues/${issueNumber}/comments`),
  ]);

  return { success: true, message: `Read issue #${issueNumber}`, data: { ... } };
}

// 3. Progress messages — shown to user while tool runs
progressMessages: {
  inProgress: (p) => `Reading issue #${p.issueNumber} in ${p.repo}`,
  completed:  (p) => `Read issue #${p.issueNumber}`,
  failed: "Failed to read issue",
}

// 4. Export as PolarisTool
export const readGithubIssue: PolarisTool = {
  name, description, inputSchema, executor, progressMessages
};
Pattern: PolarisTool Every tool, whether it queries the local workspace or calls GitHub, conforms to the same shape: a Zod inputSchema for validation, an executor function receiving (input, context), and optional progressMessages for the UI. This uniformity means the AI system (ToolRegistry, ToolExecutor, StreamingHandler) doesn't need any special handling for integration tools.

What the LLM Sees

The Zod schemas are automatically converted to JSON Schema and sent to the LLM as function definitions. The .describe() calls on each field become the parameter descriptions the model reads. For example, the LLM sees something like: 

// What the LLM's tool definition looks like:
{
  "name": "searchGithubIssues",
  "description": "Search GitHub issues across repos or within a specific repo.",
  "parameters": {
    "query":   { "type": "string",  "description": "Search query for issues" },
    "repo":    { "type": "string",  "description": "Repository in owner/repo format" },
    "state":   { "enum": ["open", "closed"] },
    "labels":  { "type": "array",  "items": { "type": "string" } },
    "perPage": { "type": "number", "default": 30, "maximum": 100 }
  }
}

The model decides when to call these tools based on the user's intent. If a user says "check if there's already a bug report for the login issue", the model will call searchGithubIssues with an appropriate query.

05

Skills: Compound Workflows

Tools are atomic operations. Skills are multi-step workflows that orchestrate multiple tools. The PR adds two skills:

Create GitHub Issue

When the LLM reads this skill (via the readSkill tool), it gets a detailed playbook:

Skill: Create GitHub Issue — Step Sequence
1. listGithubReposidentify target repo
↓
2. searchGithubIssuescheck for duplicates
↓ no duplicate found
3. listGithubLabelspick appropriate labels
↓
4. createGithubIssuecreate with title, body, labels
↓
5. Report backconfirm with issue # and link

The skill instructs the AI to always check for duplicates before creating. Without this, an eager LLM might create duplicate issues every time.

Send to Copilot

This skill assigns an existing issue to copilot-swe-agent (GitHub's AI coding agent). The workflow:

  1. Find the issue (via searchGithubIssues if needed)
  2. Verify it's open (via readGithubIssue)
  3. Assign copilot-swe-agent (via updateGithubIssue)
  4. Confirm the assignment
Tools vs. Skills Tools are functions the AI can call directly. Skills are prompt-based instructions that teach the AI how to use multiple tools together for a complex task. Skills are registered in the SkillRegistry and the AI loads them on-demand via the readSkill tool when a user's request matches a skill description. This is an "inner monologue" pattern: the AI reads the skill, then follows its steps.

How Skills are Surfaced

The system prompt includes an "Available Skills" section listing all registered skills with their titles and descriptions. When the AI recognizes a user request that matches a skill, it calls readSkill to load the full instructions. The skill body is injected into the conversation as a tool result, and the AI follows it step-by-step.

Note The tools are registered for all agents (currently just the "general" agent). If a user hasn't connected GitHub, the tools still appear in the LLM's tool list but will return helpful error messages when invoked. This avoids dynamic tool registration complexity.

The User Experience

Settings: Connecting GitHub

A new Integrations tab appears in the Settings dialog, positioned between "Calendars" and "Advanced". The page shows:

  • A header with "GitHub" and a connection status badge (green "Connected" / gray "Not connected")
  • A description: "Connect your GitHub account to list repositories, search and manage issues, and send tasks to GitHub Copilot directly from chat."
  • The WorkOS Pipes widget, which handles the OAuth flow inline

The connection status is checked via a new GET /integrations/connections endpoint that returns the status of all known providers. This endpoint checks WorkOS for each provider in parallel.

Settings UI Flow
Settings → Integrations
↓
GET /integrations/connectionschecks WorkOS for each provider
↓
Shows status badge"Connected" or "Not connected"
Pipes widgetOAuth connect/disconnect

Chat: Using GitHub Tools

Once connected, users interact naturally in chat:

  • "Show me my recent repos" → AI calls listGithubRepos
  • "Search for login bugs in acme/frontend" → AI calls searchGithubIssues
  • "File a bug for the crash on the settings page" → AI reads the "Create GitHub issue" skill and follows the multi-step workflow
  • "Assign that issue to Copilot" → AI reads the "Send to Copilot" skill

The progress messages give real-time feedback: "Searching issues in acme/frontend…" → "Found 12 issues".

Graceful Degradation

If GitHub isn't connected and the user asks to search issues, the AI gets back a clear error: "Please connect your GitHub account in Settings > Integrations." The AI relays this to the user with a direct pointer to where to set it up.

The Integration Type

The Integration type is a metadata descriptor, not a runtime construct. It exists to document what an integration provides: 

export interface Integration {
  id: string;       // "github"
  name: string;     // "GitHub"
  provider: string; // WorkOS provider name
  tools: string[];  // tool names this integration provides
  skills: string[]; // skill titles this integration provides
}

This could be used in the future for dynamic tool registration (only show GitHub tools if GitHub is connected), per-integration settings pages, or auditing which integration each tool belongs to.

Tradeoffs & Design Decisions

Static vs. Dynamic Tool Registration

Approach Pros Cons
Static (chosen) Simple, predictable. Tools always registered. Error messages guide users to connect. LLM sees tools it might not be able to use. Wastes context window tokens on tool definitions.
Dynamic LLM only sees tools it can actually use. Cleaner tool list. Requires per-request tool registration. Adds latency (check connection status before building tool list). More complex caching.
Tradeoff Analysis Static registration was the right call for a first integration. With 6 GitHub tools added to ~14 existing tools, the overhead is minimal. If Polaris grows to support 10+ integrations each with 5-10 tools, the LLM would receive 50-100 tool definitions—at that point, dynamic registration (or at least per-integration enable/disable) would become important.

WorkOS Pipes vs. Custom OAuth

Approach Pros Cons
WorkOS Pipes (chosen) Zero OAuth implementation code. Token refresh handled automatically. Pre-built UI widget. Already in Polaris (used for calendars). Vendor dependency. Less control over scopes. Additional cost per provider. Abstraction over GitHub's API.
Custom OAuth Full control. No vendor cost. Custom scopes. Significant implementation effort (redirect handling, token storage, encryption, refresh logic, revocation). Security surface area.

No Token Caching

Every tool call fetches a fresh token from WorkOS. This means an AI workflow that calls 4 GitHub tools makes 4 separate WorkOS API calls. The alternative would be caching the token for the duration of the request/session.

Why No Caching Simplicity wins here. WorkOS likely has its own internal cache. The added latency of a WorkOS roundtrip per tool call is negligible compared to the GitHub API call itself and the LLM inference time. Caching would require managing token expiry and invalidation—complexity that isn't justified by the current usage patterns.

GitHub REST API vs. GraphQL

The implementation uses GitHub's REST API v3 (via api.github.com) rather than the GraphQL v4 API. REST is simpler to wrap in individual tools (one endpoint per operation), while GraphQL would allow fetching related data in fewer requests but adds complexity in query construction.

All Tools on One Agent

All GitHub tools are added to the "general" agent. An alternative would be a dedicated "GitHub agent" that only appears when GitHub is connected. The current approach is simpler and leverages the LLM's ability to decide when to use each tool.

10

Limitations & Future Work

Current Limitations

  • Issues only. No PR support (create/review/merge), no file operations (read/write), no Actions/Workflows, no Discussions, no Projects (v2). These are all natural extensions using the same pattern.
  • No pagination. Tools return the first page of results (up to 100 items via perPage). For repos with thousands of issues, this limits discoverability.
  • No webhooks. Polaris can only pull data from GitHub. It can't react to GitHub events (new issues, PR reviews, etc.) in real-time. This would require a webhook receiver and event routing system.
  • Read-only comments. readGithubIssue fetches comments but there's no tool to post comments.
  • No org-level management. Connection is per-user, not per-organization. If a team shares a Polaris workspace, each member must connect their own GitHub account.
  • Static tool presence. GitHub tools show up in the LLM's tool list even for users who haven't connected GitHub, using tokens for tool definitions that won't be useful.

Natural Extensions

  • Pull request tools: createPullRequest, listPullRequests, reviewPullRequest, mergePullRequest
  • Comment tools: commentOnIssue, commentOnPullRequest
  • File tools: readGithubFile, listGithubDirectory for browsing repo contents
  • Other integrations: Linear, Jira, Slack, Notion—all following the same recipe from Section 8
  • Dynamic tools: Only send integration tools to the LLM when the user has connected that integration
  • Webhook support: React to GitHub events (new issues, PR reviews) and surface them in Polaris
  • Two-way linking: When creating a GitHub issue from Polaris, store a backlink in the Polaris workspace so the node links to the issue
The Bigger Picture This PR transforms Polaris's AI from a workspace-internal system to an externally-connected agent. The infrastructure it establishes—IIntegrationService, scoped token getters, the integration metadata type, the tools-and-skills pattern—is designed to scale to many integrations. The architectural investment is front-loaded; adding the next integration will be significantly simpler.
